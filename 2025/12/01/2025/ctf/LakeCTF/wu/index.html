Web | Challenge: Gamblecore

  • Dạng challenge này tác giả thiết kế mang thiên hướng RNG + bug (RNG: Random Number Generator) nên sẽ mang tính may rủi rất cao đúng như tên gọi của challenge:>
  • Phân tích qua file server.js, file xử lý chính của challenge:

  • Từ thông tin trên, ta có 6 api và tính năng của từng api:

    • /api/balance: api đưa thông tin số dư hiện tại.
    • /api/gamble: trò chơi gacha may rủi của challenge.
    • /api/convert: api hỗ trợ chuyển coins về usd.
    • /api/flag: api trao đổi flag khi có đủ 10$.
    • /api/deposit: bị disable.
    • /api/withdraw: bị disable.

  • /api/deposit + /api/withdraw bị disable.

  • Vậy chỉ còn 2 API có thể thay đổi số dư:

    • POST /api/gamble
    • POST /api/convert

  • Nên toàn bộ bài này là chơi qua lại giữa gamble và convert.

  • Phân tích hướng đi:

  • Tại /api/convert, ta thấy 1 đoạn này rất đáng nghi:

  • wallet.coins là số thực rất nhỏ lúc đầu: 10e-6 = 0.00001.

  • Trong khi đó lại dùng: const coinBalance = parseInt(wallet.coins);.

  • Hãy thử tự mở devtool của trình duyệt và test:

  • Sẽ có một hành vi rất “quái dị” của parseInt với số thực rất nhỏ.

  • Từ điều đó ta sẽ có cách để khiến coinBalance (int) lớn hơn rất nhiều so với wallet.coins thực tế.

  • Đây chính là bug.

  • Kế tiếp ta sẽ tận dụng /api/gamble để khiến wallet.coins giảm giá trị xuống bé hơn 9e-07 sau mỗi lần cược thất bại, rồi sau đó chuyển sang /api/convert để chuyển coins sang usd. Sau khi chuyển sang usd thành công thì lúc này wallet.coins sẽ âm dẫn đến việc ta không thể cược bằng coins nữa, nên ta sẽ all in tất cả usd hiện có nếu được 3 lần trở đi cược thành công liên tiếp thì ta hoàn toàn có thể đủ usd và mua được flag.

  • Ví dụ: Từ $0.01 → $0.1 → $1 → $10 chỉ cần 3 lần win liên tiếp (mỗi lần bet hết vốn).

  • Nếu fail thì buộc phải reset lại session và cược lại, lặp lại cho đến khi thành công thì dừng, vì tỉ lệ thắng 9% và cược không giới hạn (bằng reset session) nên hoàn toàn có khả năng xảy ra 3 lần cược thắng trở lên.

  • Dưới đây là script python dựng lại các ý tưởng trên.

  • Ta có thể mở nhiều terminal chạy script nhằm tăng khả năng có được flag.

Pwn | Challenge: stackception-1

1. Ý nghĩa từng file

[+] file chall.py:

  • Là keo dán toàn bộ bài:

    • Nhận một dòng input từ user.
    • Đổi _ → khoảng trắng, | → xuống dòng → ra file input.asm.

  • Gọi stackception-asm để dịch input.asm → out.bin.

  • Gọi stackception (VM) để chạy out.bin.

  • Nên mình không nói chuyện trực tiếp với VM, mà phải viết asm theo đúng format:

  • Ví dụ:
    → gửi: push_1|push_2|add

[+] file stackception-asm:

  • Là assembler: chuyển code asm của mình thành bytecode cho VM.

  • Nó hỗ trợ:

    • Lệnh dạng: push 123, add, mul, call main, jmp loop, v.v.
    • Label: main:, loop:, …

  • jmp label / call label sẽ được nó tự expand thành:push <địa_chỉ_label>

  • rồi mới jmp / call

[+] file stackception-1: (VM)

  • Đây là chỗ chạy bytecode. Có:

    • data stack (stack dữ liệu),
    • call stack (stack return address), mảng chứa bytecode (s).

  • Các opcode chính:

1. Bug & ý tưởng chính

Có hai ý chính:

(1) PUSH overflow sang code

  • Data stack và vùng code nằm liền nhau trong bộ nhớ.

  • Lệnh PUSH không check tràn:

  • Khi stack còn nhỏ → viết đúng vào vùng stack.

  • Khi stack đủ lớn → viết đè sang vùng bytecode.

Nghĩa là: dùng đủ nhiều push, ta có thể sửa chính chương trình đang chạy (self-modifying code).

(2) Một số lệnh đọc “top of stack” từ nhầm chỗ

  • Các lệnh như jmp, call, write, exit không đọc từ data stack, mà lại đọc từ một mảng khác bị chồng địa chỉ với: data stack và cả vùng code.

  • Khi stack vừa phải → vô tình vẫn đọc đúng (vì chồng lên data stack).

  • Khi stack quá lớn → nó bắt đầu đọc nhầm sang bytecode.

  • Kết quả:

    • Nhảy (jmp/call) tới địa chỉ lấy từ bytecode mình vừa overwrite.
    • exit cũng return giá trị từ bytecode.
  1. Exploit idea:

  • Dùng rất nhiều push 0 (khoảng 512 lần)
    → làm stack cao đến mức PUSH bắt đầu ghi đè từ instruction đầu tiên trong vùng bytecode.

  • Tiếp tục push để ghi đè:

    • Ghi opcode của instruction 0 thành 11 (FLAG).
    • Ghi giá trị tiếp theo thành 0 (để dùng làm “PC target” – nơi sẽ nhảy về, là instruction 0).

  • Gọi call (hoặc jmp) không có operand khi stack đang rất to:

  • Do bug, VM sẽ lấy địa chỉ nhảy từ vùng bytecode (chứ không phải từ stack “đúng”).

  • Mà ta đã sửa bytecode đó thành 0 → PC = 0.

  • Chương trình chạy lại từ instruction 0, nhưng Instruction 0 giờ là opcode 11 (FLAG).

  • VM đọc flag, in Flag: …, xong!

  • Toàn bộ exploit chỉ xoay quanh:

    • Đẩy nhiều PUSH để tràn vào vùng code.
    • Sửa opcode đầu thành FLAG,
    • Dùng bug nhảy đến code mình vừa sửa.

  • Thế là win.

  • Chạy script trên và lấy payload trả về rồi gửi lên server để lấy flag.

Flag: EPFL{stack-the-planet!}
