Web | No Quotes

Hướng giải

  • Bài này là combo SQLi (bypass no quotes) –> UNION để trả về username chứa payload → SSTI Jinja2 ở /home–> chạy /readflag để lấy flag.

1. Điểm yếu trong code

  • SQLi ở /login
  • Query được ghép thẳng:

  • WAF chỉ chặn ' và ", nên vẫn còn đường khác.

  • SSTI ở /home

  • session["user"] bị nhét vào HTML bằng %s trước khi Jinja render –> nếu user chứa {{ ... }} thì Jinja sẽ execute.

2. Bypass no quotes bằng backslash \

  • MySQL/MariaDB mặc định cho \ escape ký tự '.
  • Nếu username kết thúc bằng \, nó sẽ escape dấu ' đóng và làm câu SQL bị lệch, khiến phần password trở thành nơi để truyền SQL injection mà không cần quotes.

  • Với bài này thì có hai hướng là dùng information_schema.processlist hoặc Quine (trả về đúng với những gì đã nhập).

  • Ta UNION vào processlist của chính connection (ID = CONNECTION_ID()), rồi dùng SUBSTRING_INDEX để cắt ra đúng phần username/password từ INFO.

Poc và giải thích

1. Phần username: SSTI dotless/quoteless (Jinja)

1.1 Vì sao dict(x=1)|first?

  • Ta bị cấm quote nên không thể viết __init__, os, 'popen'…

  • Mẹo: dict(__init__=1) tạo dict có key __init__, rồi |first lấy key đầu tiên → trả về string __init__ mà không cần quotes.

  • Tương tự:
    dict(os=1)|first → “os”

dict(popen=1)|first → “popen”

dict(c=1)|first → “c”

1.2 Vì sao dùng |attr(…)?

  • Ta bị cấm dấu . nên không thể cycler.__init__....
    |attr(name) là cách truy cập attribute không cần dot.

nhưng được viết lại để: không dot, không quotes

2. UNION SELECT để trả về row giả (qua check)

  • Mục tiêu:

<col1> phải bằng đúng username đã gửi (để pass row[0]==username_input)

<col2> phải bằng đúng sha256(password_input) (để pass hash check)

2.1 Vì sao CAST(0x… AS CHAR)?

  • Không dùng quotes nên không thể {{...}}.
  • Ta sẽ đưa username ở dạng hex literal:
  • Giải nghĩa:

=> 0x7b7b...7d7d5c chính là chuỗi username {{...}}\.

CAST(... AS CHAR) ép MySQL coi hex đó như chuỗi text để trả về làm cột 1.

2.2 Phần quine: làm sao để <col2> = SHA256(password_input)?

  • Ý tưởng quine tổng quát

    • Tạo một template @q có chứa một placeholder (dùng 0x3f = ký tự ?)
    • Sau đó dựng ra chuỗi hoàn chỉnh bằng: REPLACE(@q, 0x3f, LOWER(HEX(@q)))

  • Nghĩa là: thay ? bằng hex của cả template @q.

2.4 Vì sao LOWER(HEX(@q))?

  • HEX(@q) trả chuỗi hex thường là uppercase.
  • Dùng LOWER(...) để trùng hexdigest() (lowercase).

2.5 Cấu trúc IF((@q := ... ) IS NOT NULL, ..., 0)

  • Giải thích:

(@q := 0x...) gán template vào biến session @q.

... IS NOT NULL luôn đúng (vì gán chuỗi), nên nhánh TRUE chạy.

Nhánh TRUE tính hash của chuỗi quine.

  • Đây là kỹ thuật vừa gán biến vừa dùng trong cùng một biểu thức.

2.6 Vì sao bên trong template @q lại chứa 0x3f?

  • Trong template có nhét sẵn một ? để sau đó replace bằng HEX(@q).

  • Tức là template của password injection đại khái là:

Sau khi thay, P chính là chuỗi password injection cuối cùng mà ta muốn băm.

2.7 Toàn bộ UNION SELECT trên đang làm gì?

Tóm gọn, payload trả về row giả kiểu:

  • Cột 1: CAST(0x<hex(username)> AS CHAR)
    → đúng y username input → pass check 1
  • Cột 2: LOWER(SHA2( REPLACE(@q, ?, HEX(@q)) ,256))
    → đúng y sha256(password_input) → pass check 2
